已知限制标识了此版本服务不支持或无法与其正确互操作的平台、设备或功能。



== 等待故障回复完成后再运行发现

故障转移完成后，请勿在源 vCenter 上手动启动发现。等待故障恢复完成，然后在源 vCenter 上启动发现。



== NetApp Console可能无法发现Amazon FSx for NetApp ONTAP

有时， NetApp Console无法发现Amazon FSx for NetApp ONTAP集群。这可能是因为 FSx 凭证不正确。

*解决方法*：在NetApp Console中添加Amazon FSx for NetApp ONTAP集群，并定期刷新集群以显示任何更改。

如果需要从NetApp Disaster Recovery中删除ONTAP FSx 集群，请完成以下步骤：

. 在NetApp Console代理中，使用云提供商的连接选项，连接到控制台代理运行的 Linux VM，使用 `docker restart occm`命令。


参考 https://docs.netapp.com/us-en/console-setup-admin/task-managing-connectors.html#connect-to-the-linux-vm["管理现有的控制台代理"^]。

. 在NetApp Console系统页面中，再次添加Amazon FSx for ONTAP系统并提供 FSx 凭证。
+
参考 https://docs.aws.amazon.com/fsx/latest/ONTAPGuide/getting-started-step1.html["创建Amazon FSx for NetApp ONTAP文件系统"^]。

. 从NetApp Disaster Recovery中，选择“*站点*”，在 vCenter 行上选择“*操作*”选项image:../use/icon-vertical-dots.png["NetApp Disaster Recovery中的操作菜单图标"]，然后从“操作”菜单中选择“*刷新*”以刷新NetApp Disaster Recovery中的 FSx 发现。
+
这将重新发现数据存储、其虚拟机及其目标关系。





== Google Cloud NetApp Volumes的局限性

* 运行故障转移测试后，至少需要等待 52 小时才能删除克隆卷。您必须手动删除该卷。  52 小时后，您可以再次测试故障转移。
* 如果挂载操作的任何部分失败，故障转移将不会成功，作业将超时。  Google 最多需要三天时间来调查该问题，在此期间，vCenter 上所有与数据存储相关的操作都会被阻止。
